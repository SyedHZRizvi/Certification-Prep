## 🎯 Answers + Explanations

### Q1: **B. `ip addr`**
`ip addr` (or `ip a`) replaces `ifconfig` for viewing addresses. `ip link` is for L1/L2 status. `ip route` is the routing table. `ip neigh` is ARP.

### Q2: **A. `ss -tulpn`**
`ss` (socket statistics) replaces `netstat`. Flags: `-t` TCP, `-u` UDP, `-l` listening, `-p` processes, `-n` numeric.

### Q3: **B. `firewall-cmd --get-active-zones`**
"Active" = zones with at least one interface assigned. `--get-zones` lists ALL defined zones (active or not).

### Q4: **B. add-port + --permanent then --reload**
`--permanent` writes to disk; the rule is NOT live until `--reload`. The two-step is the textbook pattern.

### Q5: **B. Root can still SSH with a key**
`PermitRootLogin no` blocks all root SSH. But if you set it to `prohibit-password` (or older `without-password`), root can still SSH with a key — the question's phrasing "PermitRootLogin no" actually blocks. Reading carefully: with `no`, root cannot SSH in *at all*. Re-reading: choice A says "Root cannot SSH in at all" — that's correct with `no`. Looking again at intent: the answer should be A. (Note: in the real exam, watch this trap — `no` vs `prohibit-password` is a frequent gotcha.) **Corrected: A is the intended answer.**

### Q6: **B. Loose permissions; StrictModes rejected the key**
The Priya bug from the reading. Check `~/.ssh` is 700 and `~/.ssh/authorized_keys` is 600. `/var/log/auth.log` will say "bad ownership or modes for directory".

### Q7: **A. `ssh -L 8080:db:5432 jump`**
**L**ocal listens **L**ocally on 8080; forwards to db:5432 via jump. `-R` would have the REMOTE listening. `-D` is SOCKS dynamic. `-J` is ProxyJump.

### Q8: **B. 700**
`~/.ssh/` must be 700 (rwx for owner only). `authorized_keys` should be 600. Anything looser triggers StrictModes rejection.

### Q9: **B. `/etc/resolv.conf`**
Holds `nameserver`, `search`, `options` directives. Often auto-managed by NetworkManager or systemd-resolved.

### Q10: **B. Manual edits are overwritten**
On systemd-resolved or NetworkManager-managed systems, /etc/resolv.conf is regenerated. Configure DNS through `nmcli` (for NetworkManager) or `/etc/systemd/resolved.conf.d/` (for systemd-resolved).

### Q11: **B. `/etc/nsswitch.conf`**
Sets the order of name sources: `hosts: files dns` means /etc/hosts first, then DNS.

### Q12: **B. `nc -vz` or `nmap -p`**
`nc -vz host 443` opens a TCP connection (no data) and reports success/failure. `nmap -p 443 host` does similar. `ping` is ICMP, not TCP.

### Q13: **B. `nmcli` or netplan YAML**
On Ubuntu Server, netplan is the default. `nmcli` works if NetworkManager renderer is configured. Editing /etc/network/interfaces is legacy (pre-netplan).

### Q14: **D. Both A and B work**
Modern `dig` accepts the `@server` in either position. The traditional syntax has `@server` after the type/name but before options.

### Q15: **B. `drop`**
`drop` zone: silently drops everything (no reply). `block` zone: rejects with ICMP. `public` allows selected services.

### Q16: **B. `sshd -t && systemctl reload sshd`**
`sshd -t` syntax-checks the config and exits silently on success. THEN reload. Skip this and a typo can lock you out.

### Q17: **D. All of the above**
`ss -lnp sport :22`, `lsof -i :22`, and `netstat -tlnp` all show the process owning port 22 (netstat may not be installed; ss is preferred).

### Q18: **B. Transparently chains through the bastion**
ProxyJump (also CLI flag `-J`) opens a connection to the bastion, then from the bastion to the final target, all in one ssh command. Replaces the older `ProxyCommand ssh -W %h:%p bastion`.

### Q19: **C. Destined FOR this host**
`INPUT` = inbound to local processes. `OUTPUT` = generated locally. `FORWARD` = passing through (when this host is a router).

### Q20: **D. Both B and A are plausible; check auth.log first**
Always check `/var/log/auth.log` (Debian) or `/var/log/secure` (RHEL) — sshd logs the specific rejection reason. Could be perms, firewall, or `AllowUsers`/`Match` rules.

### Q21: **B. `yes`**
`StrictModes yes` is the default. It refuses key auth if `~/.ssh` is world-writable or `authorized_keys` is loosely permissioned.

### Q22: **B. Per-user ForceCommand, PermitTTY, allowed methods, etc.**
`Match` blocks scope settings to a user, group, or address pattern. Common use: lock a build-bot account to a single command via `ForceCommand` + `PermitTTY no`.

### Q23: **A. `tcpdump -i eth0 -w capture.pcap 'tcp port 443'`**
`-i` interface, `-w` write to pcap, the BPF filter `tcp port 443` limits to HTTPS. Open in Wireshark for decryption (with keys) or inspection.

### Q24: **B. nftables is the kernel filter; firewalld is the high-level daemon**
On RHEL 9, the kernel-side packet filter is nftables. firewalld runs on top and translates its zones/services to nftables rules. iptables commands still work via a compatibility wrapper but ultimately program nftables.

### Q25: **A. `AllowGroups admins`**
Multiple groups can be listed space-separated. There's also `DenyGroups`, `AllowUsers`, `DenyUsers`.

### Q26: **A.**
- `PermitRootLogin no` — disables root SSH entirely
- `PasswordAuthentication no` — forces key-based
- `AllowGroups sshusers` — only group members can SSH
- `ClientAliveInterval 600` + `ClientAliveCountMax 0` — server sends one keepalive after 10 min idle; with CountMax 0, the next missed = disconnect. (Some prefer `ClientAliveInterval 300` + `ClientAliveCountMax 2` for less aggressive behavior.) The other choices use non-existent directives.
