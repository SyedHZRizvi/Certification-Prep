## 🎯 Answers + Explanations

### Q1: **B. JSON Mode = valid JSON; Structured Outputs = matches YOUR schema**
The difference is YOUR schema. JSON Mode says "valid"; Structured Outputs says "valid AND conforms."

### Q2: **B. Tool use with `tool_choice={"type": "tool", "name": "..."}`**
Anthropic's idiom. The forced tool call IS the structured-output mechanism.

### Q3: **B. instructor**
By Jason Liu. The de facto cross-provider Pydantic wrapper.

### Q4: **B. Only outputs one of those values for that field**
Literal/Enum are categorical constraints. Hallucinated 4th values become impossible.

### Q5: **B. Field order conditions generation (CoT in schema form)**
Structured outputs honor Pydantic field order. Reasoning-before-answer = "think first."

### Q6: **B. Somewhere in the prompt**
OpenAI's intentional safeguard. Forgetting throws an API error.

### Q7: **B. Move to L1 or L2, eliminates the failure entirely**
String replace is a band-aid. Structure guarantees eliminate the class.

### Q8: **B. Field validator raises → instructor reprompts with error context**
The closed-loop self-correction pattern.

### Q9: **B. Core part of the prompt, model uses them to route**
Anthropic explicitly: "tool descriptions ARE prompt engineering."

### Q10: **B. Lower latency via parallel dispatch**
Multi-tool turns reduce round trips.

### Q11: **B. Flatten OR multi-step**
Deeply nested schemas hurt model coherence. Decompose.

### Q12: **B. Libraries for constrained generation**
Outlines (by dottxt.ai), Guidance (by Microsoft). Critical on open-source.

### Q13: **B. L0 → L2 + product validator + lookup_sku tool**
The case-study fix. All four pieces matter.

### Q14: **B. Guaranteed parseable / schema-conforming output**
The category of runtime errors is what's eliminated.

### Q15: **B. Part of the JSON Schema, improves accuracy**
Descriptions are visible to the model. Spending 5 seconds writing one pays for itself.

### Q16: **B. Forced into a token it didn't want**
Real but rare trade-off. Quality usually comes out fine; eval to be sure.

### Q17: **B. `int = Field(..., ge=1, le=5)`**
The constraint enforces the range at validation time. Schema declares the bound.

### Q18: **B. Triggers retry with error context**
That's the whole point of instructor's `max_retries`.

### Q19: **B. OpenAI Structured Outputs**
Both are schema-enforced JSON. Just different vendor syntax.

### Q20: **B. Truncation at max_tokens producing invalid partial JSON**
A failure mode that prompt-only doesn't really have (since it never claimed validity anyway).

### Q21: **B. Make field Optional**
Don't ask for what the model doesn't have. The schema must be satisfiable.

### Q22: **C. `{"type": "tool", "name": "..."}`**
The forced-call syntax. "auto" lets the model decide; "none" forbids; this forces a specific tool.

### Q23: **B. Specific descriptions improve routing accuracy**
The model reads them. Specificity = better routing.

### Q24: **B. Field validator + retry**
Schema fixes syntax; validators fix semantics. Use both.

### Q25: **C. Bypasses rate limits**
False. Rate limits are vendor-side. instructor doesn't and can't bypass them.

### Q26: **B. L2 + Pydantic + Literal/Enum + validators + retries + Optional + T=0**
The textbook medical-extractor architecture. Every clause matters in a regulated domain.
