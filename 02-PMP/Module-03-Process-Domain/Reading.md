# PMP Module 3: Process Domain ⚙️

> **Why this module matters:** **50% of the exam.** This is HOW projects get managed — scope, schedule, cost, integration, change control. Master this and the math/calculation questions follow.

---

## 🏗️ A Story: Building A Restaurant

Maria is opening a restaurant. Phase 1: lock the location. Phase 2: design. Phase 3: build. Phase 4: hire. Phase 5: open.

She has:
- A **scope** (what she's building)
- A **schedule** (when each phase happens)
- A **budget** (how much each costs)
- **Risks** (suppliers may be late)
- **Quality** (health code compliance)
- **Communications** (with contractors, staff, marketing)

Each of these is a **management process**. Together, they're the Process Domain.

## 🧭 The 5 Process Groups (Old School, Still Relevant)

Even though the new PMP focuses on Performance Domains, the **5 Process Groups** still appear:

```
   INITIATING → PLANNING → EXECUTING → MONITORING & CONTROLLING → CLOSING
       ↓           ↓           ↓                 ↓                    ↓
     Charter   Subsidiary  Deliverables     Reports/Variance      Lessons
                Plans                                              Learned
```

🎯 These aren't sequential phases — they're **logical groupings**. M&C runs throughout.

## 🎯 Project Charter (Initiating)

The Charter is the document that **formally authorizes** the project.

### Contents:
- Project purpose
- Measurable objectives
- High-level requirements
- Assumptions and constraints
- High-level risks
- Summary milestones
- Summary budget
- Approval requirements
- Project Manager assignment + authority
- Sponsor name + authority

### Who Signs:
- Project Sponsor (NOT the PM)
- The PM assists in creating it.

🎯 **Exam tip:** "What document gives the PM authority?" → **Project Charter.**

## 📋 Scope Management

### Define Scope:
- 📋 **Project scope** = work to be done
- 🎁 **Product scope** = features & functions of the deliverable

### Key Documents:
- **Project Scope Statement** — detailed scope, deliverables, exclusions
- **WBS (Work Breakdown Structure)** — hierarchical decomposition
- **WBS Dictionary** — details for each WBS item
- **Scope Baseline** = Scope Statement + WBS + WBS Dictionary

### WBS Rules:
- Each level breaks down to smaller pieces
- "Work packages" = lowest level (typically <80 hrs of work)
- Captures **100% of project scope** (100% rule)
- Don't include activities/tasks (those go in the schedule)

### Scope Creep vs Gold Plating
- **Scope creep** = uncontrolled changes adding scope
- **Gold plating** = team adds extras without authorization
- Both BAD. Use change control.

🎯 **Exam pattern:** "Stakeholder requests new feature mid-project."
- ❌ Just add it
- ✅ **Analyze impact, run through change control process**

## 📅 Schedule Management

### Key Steps (in order):
1. **Define activities** — what work needs doing
2. **Sequence activities** — dependencies
3. **Estimate durations** — how long
4. **Develop schedule** — combine into a timeline
5. **Control schedule** — monitor & adjust

### Estimation Techniques:
- **Analogous (Top-down)** — based on similar past projects (fast, less accurate)
- **Parametric** — based on math relationships (e.g., $/sq ft)
- **Three-point (PERT)** — Optimistic, Most likely, Pessimistic
  - PERT formula: **(O + 4M + P) / 6**
  - Standard deviation: **(P - O) / 6**
- **Bottom-up** — sum of detailed estimates (most accurate, slow)

### Critical Path Method (CPM)
The **Critical Path** is the **longest path** through the project network.
- Determines the **shortest possible duration**
- Tasks on critical path have **zero float (slack)**
- Delays on critical path delay the project
- Float (slack) = how long a task can slip without delaying the project

🎯 **Exam pattern:** "Task A has 5 days of float. Task A is delayed by 3 days. Project end date?"
- Answer: **Same.** 3 < 5 means no impact on critical path.

### Schedule Compression
When you need to shorten the schedule:

| Technique | What It Does | Cost? | Risk? |
|-----------|--------------|-------|-------|
| **Crashing** | Add resources to reduce duration | 💰 More cost | Limited risk |
| **Fast-tracking** | Run activities in parallel | 🆓 Often free | 🚨 More risk |

🎯 **Exam pattern:** "Need to finish faster, no extra budget." → **Fast-tracking.**
"Need to finish faster, willing to spend more." → **Crashing.**

## 💰 Cost Management

### Steps:
1. **Plan cost management** — how to estimate/budget/control
2. **Estimate costs** — for each activity
3. **Determine budget** — aggregate into baseline
4. **Control costs** — monitor & adjust

### Estimation Types:
| Type | Range | When |
|------|-------|------|
| Rough Order of Magnitude (ROM) | -25% to +75% | Initiating |
| Budgetary | -10% to +25% | Planning |
| Definitive | -5% to +10% | Late planning |

### Cost Baseline
- The approved budget (without management reserves)
- **Time-phased** — shows when money is spent
- Used as the EVM measurement baseline (Module 6!)

### Reserves
- **Contingency Reserve** — for KNOWN risks ("known unknowns") — part of cost baseline
- **Management Reserve** — for UNKNOWN risks ("unknown unknowns") — NOT part of baseline; sponsor controls

🎯 **Exam pattern:** "PM identifies a new risk not in the risk register. They need to spend $5K to mitigate."
- ❌ Use Contingency
- ✅ **Request from Management Reserve** (it's an unknown unknown).

## 🔄 Integration Management — The Big Picture

The PM's primary responsibility. Coordinates ALL 10 knowledge areas.

### Key Documents:
- **Project Charter** — authorization
- **Project Management Plan** — the master plan (combines all subsidiary plans)
- **Project Document Updates** — keep things current
- **Approved Change Requests** — what's been approved to do

### Change Control Process:
1. ✅ Receive change request
2. 🔍 Evaluate impact (scope, schedule, cost, etc.)
3. 🤝 Submit to **Change Control Board (CCB)**
4. ✅ Approve / Reject
5. 📋 Update plan
6. 🛠️ Communicate

🎯 **Exam pattern:** "Stakeholder demands a scope change."
- ❌ Reject without analysis
- ❌ Just add it
- ✅ **Evaluate impact, submit to CCB, follow through.**

---

## 📊 Baselines (Critical Concept!)

Three baselines that get **frozen** at planning:

1. **Scope baseline** = Scope Statement + WBS + WBS Dictionary
2. **Schedule baseline** = approved schedule
3. **Cost baseline** = approved budget (without management reserve)

Together = **Performance Measurement Baseline (PMB)**.

🎯 To change a baseline → MUST go through change control.

## 🏁 Project Closure

### Steps:
1. ✅ Confirm work is done per scope
2. ✅ Get formal acceptance
3. 📚 Document lessons learned
4. 📦 Archive records
5. 🎉 Release resources
6. 🏆 Celebrate

### Lessons Learned
- Captured throughout the project (not just at end)
- Stored in OPAs for future projects
- Both successes and failures

🎯 **Exam pattern:** "Project just ended. PM realizes a major issue wasn't documented."
- ❌ Skip it
- ✅ **Document in lessons learned for future projects.**
